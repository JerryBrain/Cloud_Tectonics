# 云构模组 (Cloud Tectonics) - 开发与架构说明文档

本文件旨在详细描述“云构模组”重构后的多平台（Architectury）项目结构、技术路线、已实现功能以及对应的代码文件位置，方便后续进行手动开发或与其他 AI 编码助手配合作业。

---

## 1. 项目基本信息
- **游戏版本**：Minecraft 1.20.1
- **构建工具**：Gradle 8.4
- **模组架构**：Architectury Multi-loader (Common, Forge, Fabric)
- **依赖版本**：
  - Minecraft Forge: `47.4.20`
  - Fabric Loader: `0.15.6`
  - Fabric API: `0.91.0+1.20.1`
  - Architectury API: `9.2.14`
  - Java SDK: `JDK 17` (配置支持 JDK 21 运行环境)
- **主要特性**：参数化古建生成（大木作）、思考（Ponder）风格 3D 营造编辑器、手持虚影预览与 15° 步长任意角度旋转、基于 VBO 烘焙的 O(1) Draw Call 高性能客户端渲染、以及物理碰撞代理方块系统。

## 3. 技术路线与多端协同机制

本架构的核心在于**“核心逻辑完全通用化，平台差异利用 ExpectPlatform/适配器 桥接”**。

### 3.1 跨平台适配与桥接技术 (ExpectPlatform)
*   **注册差异处理**：
    - 通用注册类 [ModBlocks.java](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/common/src/main/java/com/example/cloudtectonics/registry/ModBlocks.java) 声明了带 `@ExpectPlatform` 注解的方法 `createProxyBlock`。
    - **Forge 编译分支** 会指向 [ModBlocksImpl.java (Forge)](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/forge/src/main/java/com/example/cloudtectonics/registry/forge/ModBlocksImpl.java)，返回搭载动态声音逻辑的 [ForgeBuildingProxyBlock.java](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/forge/src/main/java/com/example/cloudtectonics/block/ForgeBuildingProxyBlock.java)。
    - **Fabric 编译分支** 会指向 [ModBlocksImpl.java (Fabric)](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/fabric/src/main/java/com/example/cloudtectonics/registry/fabric/ModBlocksImpl.java)，直接返回 vanilla 的通用 [BuildingProxyBlock.java](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/common/src/main/java/com/example/cloudtectonics/block/BuildingProxyBlock.java)。

### 3.2 客户端网络包同步兼容 (`onDataPacket`)
*   `onDataPacket` 是 Forge 独有用来处理客户端数据块更新同步的方法。在 `common` 模块中不能直接带 `@Override` 覆写。
*   **解决方案**：在 [BuildingAnchorBlockEntity.java](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/common/src/main/java/com/example/cloudtectonics/blockentity/BuildingAnchorBlockEntity.java#L327) 中写成通用的 Java 方法（省略了 `@Override` 并去掉 `super.onDataPacket` 的依赖）。这使得它在 Fabric/Vanilla 编译时只会被当做一个普通方法（不产生符号错误），但在 Forge 运行时会完全契合签名从而实现自动重载，更新并标记客户端 VBO 静态烘焙渲染。

### 3.3 视口渲染与 O(1) Draw Call (VBO)
*   古建木结构庞大（动辄数千个构件立方体），为了防止 CPU 向 GPU 频繁传递顶点引发卡顿，[BuildingAnchorRenderer.java](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/common/src/main/java/com/example/cloudtectonics/client/BuildingAnchorRenderer.java) 会在方块实体生成或数据改变时，一次性把全部几何体烘焙进一个 GPU 显存段（`VertexBuffer`）。每一帧绘制只需调用单次 `vbo.drawWithShader(...)`。
*   同时在顶点绘制中，对于拉伸的构件平面，渲染器按 `1.0f` 的栅格进行**平铺单元细分**，重新映射 UV，完美解决了贴图模糊越界及闪烁花屏的问题。

### 3.4 物理代理方块机制与 OBB SAT 判定
*   当玩家确定建造后，服务端根据几何边界铺设物理代理方块 [BuildingProxyBlock.java](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/common/src/main/java/com/example/cloudtectonics/block/BuildingProxyBlock.java)。
*   为了防止 15° 或其他非直角倾斜构件产生大面积空气墙，[GomedricTransformer.java](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/math/GomedricTransformer.java) 引入了**分离轴定理 (SAT) 判定系统**，只有在倾斜构件有向包围盒（OBB）物理占用的 3D 世界体素格中，才会真正留存并激活物理代理方块，缩减了 80% 的多余实体开销并杜绝了空气墙。

### 3.5 1/16 像素级精度对齐与“材份制”模块比例系统
*   **亚像素格网对齐**：在 [StructureComponent.java](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/common/src/main/java/com/example/cloudtectonics/math/StructureComponent.java) 中实现强制的 $1/16$ 格网四舍五入。所有几何组件的局部偏移坐标 `localPos` 和几何大小 `size` 都会在初始化时被规整到 `0.0625` 的整倍数上，从而保证了生成的 3D 模型细节完全符合 Minecraft 像素级精度，排除了浮点微小偏差导致的拼接缝隙。
*   **营造“材份”模块比例**：在 `BuildingConfig` 中定义了动态的 `caiSize`（一材）与 `fenSize`（一分，一材折合 15 分）比例标准。所有核心构件如立柱直径 ([ColumnFactory](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/common/src/main/java/com/example/cloudtectonics/math/ColumnFactory.java))、梁架横截面 ([BeamFactory](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/common/src/main/java/com/example/cloudtectonics/math/BeamFactory.java)) 和铺作斗拱 ([DougongFactory](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/common/src/main/java/com/example/cloudtectonics/math/DougongFactory.java)) 的尺寸参数完全基于材/分模块换算，使整栋古建大木作能够按照经典的《营造法式》规则进行无级缩放。
*   **骨架节点树 (Skeleton Node Graph) 对齐**：在 `BuildingConfig.buildSkeletonNodes()` 中统一管理并预计算各构件的基准三维空间节点（如立柱顶部节点、各檩条中心节点、正脊中心点）。生成梁、檩、椽、斗拱等构件时直接挂接到对应的骨架节点上进行拉伸或对齐，从物理上杜绝了修改参数时产生“穿模”与“悬空”漏洞。
*   **双层重檐与多层楼阁堆叠 (Multi-story Structure)**：在 `ParametricBuildingGenerator` 中引入垂直链式 `stories` 循环堆叠。每层建筑根据 `config.baseY` 进行 Y 轴偏移，并在各层之间铺设木板平台（Floor Deck）与侧面腰檐，支持各层柱网向上层层收缩（收分），大幅度增强了古建的垂直可扩展度。
*   **Wedge（三角斜切面）在“昂”结构中的渲染**：在斗拱工厂中将“昂身”与“昂嘴”重构为了 `ShapeType.WEDGE` 三角斜切几何体，并在 [`BuildingAnchorRenderer.buildWedgeQuads`](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/common/src/main/java/com/example/cloudtectonics/client/BuildingAnchorRenderer.java) 渲染器中使用折叠 degenerate 顶点的四边形拼接算法，以单次 O(1) 的 VBO Draw Call 绘制出高颜值的斜向挑出形态。

## 6. 未来架构重构与特性演进方向

### 6.1 [已于本阶段完成] 骨架节点驱动与“材份制”模块化系统 (Modular Assembly & Skeleton Node Graph)
- **已实现 - 骨架节点树对齐**：不采用散落的坐标偏移算法，已建立了中央 `SkeletonNodeGraph`，使梁架、檩条等构件的坐标与长度完全从骨架节点推导，参数改变时实现无缝对齐。
- **已实现 - 材份制大木作比例**：定义了 `caiSize` 与 `fenSize` 模块单元，所有柱、梁、铺作尺寸完全与材/分挂钩，实现整栋古建按营造比例精细缩放。
- **已实现 - 双层重檐多层叠放**：实现了垂直 Story 层叠与上层收分（收缩柱网），层间自动拼装楼板平台与挂接腰檐。
- **已实现 - 像素级精度三角昂结构**：支持在斗拱中采用 **Wedge 楔形斜切几何图元**，绘制出高颜值的斜向挑出“昂身”与“昂嘴”。

### 6.2 构件级局部破坏与保留系统 (Component-Level Partial Destruction & Preservation)
- **实现局部拆除**：在 `StructureComponent` 中引入生命值或销毁标记。当玩家左击挖掘某个位置 of 物理代理方块（`BuildingProxyBlock`）时，拦截破坏事件（阻止整栋建筑直接销毁的默认行为）。
- **高精视线求交检测 (Raycasting)**：获取玩家视角射线，与该格内叠加的所有构件包围盒（AABB）进行实际相交点判定，精确识别出玩家指向的具体子构件（例如只拆除了最外面的瓦片，而瓦片底下的椽条依然保留）。
- **动态 VBO 重绘**：标记破坏的组件为已销毁，更新客户端 VBO 剔除该组件渲染。只有当某格内的所有构件均被破坏后，才拆除对应的物理代理方块。

### 6.3 高精度亚像素级吸附式内饰系统 (High-Precision Snapping Interior Decoration System)
- **虚拟内饰子组件**：在 `BuildingAnchorBlockEntity` 中使用专门的 `List<DecorationComponent>` 存储家具，其坐标支持完全自由的浮点数三维局部坐标，不占用原版独立方块空间，可实现多内饰共用一格。
- **物理表面精准吸附 (Surface Snapping)**：放置家具时，利用射线检测出古建构件表面（如地面顶面、墙体侧面）的真实几何相交点 $(X,Y,Z)$，将家具高度吸附对齐到该表面（如贴墙壁挂、地面摆放），并支持任意角度旋转。
- **亚像素碰撞箱融合 (VoxelShape Union)**：在代理方块的 `getCollisionShape` 中收集并拼合当前格内所有内饰组件的 AABB 形成复杂的融合 `VoxelShape`，提供精度高达 $1/16$ 格网（或更高）的实体踩踏碰撞体验。

### 6.4 牌匾对联文字与挂画自定义系统 (Plaques, Couplets & Hanging Scrolls Customization System)
- **预设边框模型与文字区域**：定义标准规格的 3D 模型（如 1x3, 1x5 等规格的牌匾、对联竹简、石碑额），并在模型表面标记好特定的 2D 编辑区域（Quads 材质表面），作为文字或画作的贴图投射媒介。
- **PPT式文字排版与 Java AWT 动态绘制**：在客户端利用 Java 的 `java.awt.Graphics2D` 在内存中创建高分辨率（如 256x128, 512x256, 甚至是 1024x512 像素）的 `BufferedImage`：
  - 允许玩家在 UI 中输入内容，并像 PPT 文本框一样自由设置：横排/竖排、对齐方式、字体大小、字间距、行间距、字体阴影、描边金边以及背景底纹。
  - 支持玩家将外部字体文件（`.ttf` / `.otf`）放入游戏 `config/cloudtectonics/fonts/` 目录，模组动态加载这些字体，并在内存中进行反走样高精渲染，最后将 `BufferedImage` 上传至 Minecraft 客户端的 `DynamicTexture`（动态纹理），渲染到模型上，呈现极具中式书法美感的文字。
- **自定义挂画导入与自适应裁剪**：玩家可将本地图片（PNG/JPG）放入 `config/cloudtectonics/paintings/` 目录：
  - 模组读取图片后，使用 Java 2D 图形学引擎，根据挂画画幅（如立轴、横幅、团扇、屏风）的宽高比进行**自动缩放与自适应裁剪**（类似于网页 CSS 的 `object-fit: cover` 效果），防止图片拉伸变形。
  - 上传至动态纹理绑定，使玩家可以在古建中自由悬挂高清壁画。

### 6.5 专属古建世界区域编辑器系统 (Specialized World Editor)
由于古建全部数据由控制原点的单个方块实体参数决定，对其开发区域编辑功能具备天然的轻量化优势：
- **参数化一键克隆 (NBT Clone)**：无需像传统 WorldEdit 一样拷贝成千上万个网格方块。只需手持“建筑棒” Shift+右击 代理方块，读取 `BuildingAnchorBlockEntity` 的完整参数 NBT 并缓存在道具上，右击别处即可带 3D 虚影一键完成大殿的复制和粘贴。
- **无卡顿平移微调 (Shift/Nudge)**：当玩家使用编辑按键进行“平移 1 格”操作时，服务器先清空旧坐标下的锚点与全部代理方块，随即在新坐标生成控制锚点并写回参数 NBT，客户端 VBO 瞬间重绘，平移过程在 1 帧内无感完成。
- **镜像与对称 (Mirroring & Symmetry)**：遍历 `components` 列表，将 `offsetX` 及局部 `rotationY` 取反，支持玩家在 GUI 中一键将“左偏殿”生成并另存为对应的“右偏殿”预设。
- **撤销/重做历史栈 (Undo & Redo)**：服务器端为每个玩家维护 `Stack<UndoRecord>` 历史操作栈，不仅记录锚点的坐标和 NBT 参数，还记录放置前该区域原版的方块状态，为编辑操作提供强力“反悔药”防压碎地形。

### 6.6 免指令轮盘与 HUD 快速交互系统 (Radial Menu & HUD Dock)
为提高 UX 交互效率，避免繁杂的手打命令：
- **玻璃拟态环形轮盘 (Radial Menu)**：玩家手持建筑棒按住特定按键（如 `V` 键），屏幕中央弹出圆形半透明磨砂扇区轮盘，根据鼠标指针与中心坐标的极坐标夹角 $\theta = \operatorname{atan2}(\Delta y, \Delta x)$ 点亮相应扇区，释放按键即刻执行克隆、旋转、镜像或撤销指令。
- **浮动 Dock 工具栏**：手持建筑棒时，游戏底部常驻 Mac 风格小工具栏，支持快捷键 1~4 或鼠标点击直观进行模式切换。
- **无 UI 组合快捷键**：手持工具看地面虚影时，方向键 Up/Down/Left/Right 轴向平移 1 格，PageUp/PageDown 微调 Y 轴高度。单滚轮旋转 15°，Ctrl+滚轮旋转 1°，Shift+滚轮旋转 90°。

### 6.7 构件组合与内饰预制件包系统 (Component Grouping & Prefab)
为了实现家具组合（如桌椅套件）及构件群组（如梁架斗拱组）的整体调用与变换：
- **平铺组标识法 (Flat Prefab Grouping)**：在 [StructureComponent.java](file:///c:/Users/jyh/Desktop/coding/Cloud_Tectonics/common/src/main/java/com/example/cloudtectonics/math/StructureComponent.java) 中增加 `groupId` 与局部相对坐标。在 `prefabs.json` 注册表定义桌子椅子的相对布局，在放置组合包时通过矩阵变换旋转算法公式在大殿放置中心平铺展开（Flatten）并混入 `components` 列表。
- **嵌套树状图法 (Nested Transform Tree)**：对 `StructureComponent` 进行树状化定义（包含 leaf 与 branch 节点类型），渲染器与碰撞系统采用递归矩阵解算器（在 `PoseStack` 类似栈中叠加 Group 相对偏移），子构件自动继承父 Group 的空间位移。
